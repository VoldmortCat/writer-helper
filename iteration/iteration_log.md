## 迭代1：文本类型自动检测

### Step 1 — 描述具体痛点

用第一版 review 分析"学习/绩点/竞赛"那篇经验分享时，"人物"维度完全用不上，
因为议论文里没有人物的概念。分析时只能手动把"人物"替换成"论点与论据"，
但这意味着每次分析议论文都要人工处理一个废掉的维度。

### Step 2 — 量化影响

- 使用场景中有约 50% 是非叙事类文本（议论文、经验分享、公众号文章、课程作业）
- 面对这些文本，"人物"维度给的反馈全是无效的（"本文没有传统人物"）
- 用户需要等 AI 跑完 5 个维度，其中 1 个维度白跑，浪费 20% 的 token 和等待时间

### Step 3 — 假设原因

初始设计时默认用户写的是小说/故事类内容，所以 5 个维度都围绕叙事展开。
但实际使用中，用户也大量写议论文、经验分享、公众号推文——这些文本的核心是
"观点+论据"而不是"人物+情节"。分析框架没有适配不同文本类型的能力。

### Step 4 — 实现方案

在 `writing_buddy.py` 中增加文本类型自动检测模块：

1. **特征词表**：定义叙事类特征词（他说、走了、突然、引号对话）和议论类特征词
   （我认为、首先、因为、综上所述）
2. **评分算法**：统计两类特征词出现频率，计算比例，判断文本属于 narrative /
   argumentative / mixed
3. **两套 System Prompt**：叙事类保留原 5 维度（结构/人物/细节/节奏/语言），
   议论文将"人物"替换为"论点与论据"（结构/论点与论据/细节/节奏/语言）
4. **混合类**：当叙事和议论特征接近时，使用适配两个维度的混合分析 prompt

### Step 5 — 评估效果

- ✅ 同一篇议论文先用旧版 review（"人物"维度输出"本文没有传统人物"），
   再用新版 review（自动识别为 argumentative，"论点与论据"维度指出"依琳师姐没展开"）
- ✅ 叙事类文本不受影响，仍使用原五维度
- ✅ 超短文本（<100 字）倾向于归类为议论文，不影响使用
- ⚠️ 潜在问题：特征词检测基于规则而非语义，复杂文本（如夹叙夹议的散文）可能误判
- ⚠️ 后续改进方向：可以在 prompt 中让 AI 自己也判断文本类型，双重确认

## 迭代1（修正）：一致性检查 — AI检查前后矛盾

### Step 1 — 描述具体痛点

写作搭子的核心功能之一应该是检查长篇写作的一致性——人物设定有没有矛盾、
前面埋的伏笔后面收没收、时间线有没有冲突。
但当前只有 review（单段反馈）和 brainstorm（卡壳方向），
写长篇时作者自己很难发现前文和后文的矛盾，手工翻找极其耗时。

### Step 2 — 量化影响

- 写超过 3 章后，手动回溯检查一章需要 10-15 分钟
- 长篇写完才发现人物设定矛盾（如第一章黑发第五章棕发），
  需要整章重写，代价巨大
- 伏笔忘记回收是长篇写作最常见的 Bug，作者自己很难注意到

### Step 3 — 假设原因

初始设计只考虑了"单篇反馈"场景，没有考虑"跨章节检查"。
review 模式每次只给一段文字反馈，无法感知多章之间的对照关系。
需要一个新的检查维度——不是分析写得好不好，而是对比各章之间有没有矛盾。

### Step 4 — 实现方案

新增 `scripts/consistency_check.py`：

1. **多文件输入**：支持 `--files`（逗号分隔）和 `--dir`（目录扫描）
2. **四类检查**：
   - 🚨 **人物设定矛盾**：外貌/性格/能力/关系变化
   - ⚠️ **情节逻辑漏洞**：时间线/因果关系
   - 📌 **伏笔跟踪**：已回收 vs 未回收
   - ✅ **细节前后照应**：地名/人名/物品描述
3. **两级输出**：`--detail` 出完整报告，不传只出摘要
4. **低温度 (0.3)**：一致性检查需要严谨比对，不需要创意

### Step 5 — 评估效果

测试方式：用真实长篇选段做人工验证

| 检查项 | 测试文本 | 结果 |
|--------|---------|------|
| 人物设定矛盾 | "宗教效应" + 之前"艺术的见解"对比 | 两篇独立文章，无人物矛盾，报告如实验证 ✅ |
| 情节逻辑 | 批量测试需更多素材 | ⏳ 待测 |
| 伏笔跟踪 | 同上 | ⏳ 待测 |
| 细节照应 | 同上 | ⏳ 待测 |

结论：一致性检查可以正常扫描并输出四类报告，
但需要真实长篇写作数据才能充分验证人物矛盾和伏笔回收的识别准确率。

### 实证数据：review 模式帮用户改进写作

用户用自己的「宗教效应」议论文测试 review 模式，根据反馈改了一版：

| 对比项 | 改前 | 改后 |
|--------|------|------|
| 概念定义 | 无定义锚定，直接讲机制 | 开篇锚定「我称它为宗教效应，不是比喻」 |
| 论证结构 | 「实质：」一段塞满 4 个环节 | 拆成 3 段：造物→宣发滚雪球→资源反哺 |
| 语言准确度 | 3 处语病（再/优越/诚如） | 全部修正 |
| 结尾 | 突然截止在「归属感」 | 追加宗教 vs 消费品的对比性结论 |
| 字数 | 608 字 | 793 字 |

✅ 用户反馈：4 条修改建议全部采纳，修改后文章逻辑更清晰、结尾更有力。
证明 review 模式对真实写作有可验证的改进效果。

## 迭代1 — 额外痛点修复（v1.2）

基于第1-2轮实际使用反馈，在迭代1基础上追加修复 3 个使用体验痛点：

### 痛点1：brainstorm 给完方向后缺少"怎么开始写"指引

| 步骤 | 内容 |
|------|------|
| Step 1 痛点 | brainstorm 输出 3 个方向后直接结束，用户追问"怎么延续"没有自然承接 |
| Step 2 影响 | 每次 brainstorm 后用户要自己再想"那我到底写什么"，使用体验断崖式下降 |
| Step 3 原因 | prompt 只要求给出方向，没有要求给"下一步动手"的引导 |
| Step 4 方案 | SYSTEM_BRAINSTORM 末尾增加"下一步建议"段落，要求针对最可能选的方向给具体开头 |
| Step 5 效果 | ✅ 新增"下一步建议"段落，用户看完可以直接动手写 |

### 痛点2：review 没有"改完再跑一轮"的对比机制

| 步骤 | 内容 |
|------|------|
| Step 1 痛点 | review 提完问题就结束了，用户改完后无法自动对比前后差异 |
| Step 2 影响 | 改得有没有效果靠感觉判断，无法量化 |
| Step 3 原因 | 设计时只考虑了单次分析，没考虑修改后的验证环节 |
| Step 4 方案 | 新增 `review --compare 旧版.md 新版.md` 对比模式，自动分析改动有效性和新问题 |
| Step 5 效果 | ✅ 对比模式可用，"宗教效应"修改版已验证效果 |

### 痛点3：长文本超过 token 没有分段提示

| 步骤 | 内容 |
|------|------|
| Step 1 痛点 | 用户写超长文章时脚本静默截断或返回不完整结果，无任何提示 |
| Step 2 影响 | 不知道被截断了，读到的反馈基于不完整文本 |
| Step 3 原因 | 没有入口长度检查，直接丢给 API |
| Step 4 方案 | `check_text_length()` 函数估算 token 数，超过 MAX_TOKENS(6000) 时给出警告和分段建议 |
| Step 5 效果 | ✅ review / brainstorm / compare 三个入口全覆盖 |

## 迭代2：风格学习 — 学你的写作风格，保持一致性

### Step 1 — 描述具体痛点

写了一段时间后发现一个问题：同一个人的文章，前后风格不一致。
比如平时用短句、爱括号补充、敢用狠词的人，换了场景写东西就变成套话长篇，
读起来不像一个人写的。DeepSeek 默认输出也容易有"AI 腔"，
需要手动调整才能保持个人风格。

### Step 2 — 量化影响

- 写新类型文章时（如议论文 → 经验分享），风格容易跑偏
- 别人读你的文章会觉得"不像你写的"
- 每次都要手动回顾旧文找风格参考，耗时且不系统

### Step 3 — 假设原因

没有工具记录和对比写作风格特征。风格是一种感觉，
没有量化手段去判断"这句是不是我写的"。

### Step 4 — 实现方案

新增 `scripts/style_learner.py`，两阶段：

**第一阶段：learn — 提取风格特征**
- 统计特征（本地）：字数、句长、段长、高频词、人称比例、标点习惯
- AI 分析特征（调用 API）：用词偏好、句式特征、语气态度、修辞习惯
- 保存为 JSON 文件到 `references/styles/`

**第二阶段：check — 风格一致性检查**
- 统计对比：目标文本 vs 参考文本的量化指标差异
- AI 分析：判断目标文本是否"像"参考文本的风格
- 标记具体段落和差异点

### Step 5 — 评估效果

测试方式：用户提供4篇参考文章 -> 风格提炼 -> 识别一篇非本人文章 -> 用风格写新内容

| 测试项 | 输入 | 结果 |
|--------|------|------|
| 风格提炼 | 宗教效应 + 艺术的见解 + 网友见解 + 个人感想 | ✅ 提炼8个维度，保存为 references/styles/风格记录_用户.md |
| 风格识别 | 凌晨三点的便利店（非本人） | ✅ 正确判断"不是用户写的"，并标注具体差异：括号使用、口语狠词、语气 |
| 风格写作 | 用用户风格写"艺术之争总览" | ✅ 完成，含括号补充、口语词、直接态度 |
| 风格文件 | 持续维护 | ✅ references/styles/风格记录_用户.md |

### 与迭代1的关系

迭代1检查内容一致性（人物/情节/伏笔），迭代2检查风格一致性（语气/用词/句式）。
两个维度互补：迭代1管"写对了没有"，迭代2管"像不像你写的"。
在实际写作流程中，先跑 review 改内容，再跑 style check 保风格。

## 迭代3：个性化灵感 — 基于你之前写的内容给方向建议

### Step 1 — 描述具体痛点

brainstorm 给的方向是通用的——"你可以往左走、往右走、往回走"，
但这些方向跟用户的写作历史、立场、风格没有关系。
用户想要的是"如果你是我，以我过去的观点和风格，这个题我会怎么写"。

### Step 2 — 量化影响

- brainstorm 的 3 个方向每次都要用户自己判断"哪个方向适合我"
- 缺乏个性化的方向建议，用户拿到后还要自己再加工
- 风格记录（迭代2产出）和之前的写作存着但没有被用来生成内容

### Step 3 — 假设原因

brainstorm 设计时没有接入用户的历史写作和风格记录，
每次都是独立调用 prompt 生成通用方向。

### Step 4 — 实现方案

流程改为多步联动：

1. 用户给话题或卡壳场景
2. 读取 style/风格记录_用户.md（迭代2产出）
3. 回顾用户之前写的相关文章（data/ 中已有）
4. 分析用户之前的立场、关注的话题、论证风格
5. 给 2-3 个符合用户风格和立场的发展方向，每个方向附带一段示例开头
6. 用户选一个接着写，或换方向

### Step 5 — 评估效果

测试方式：给一个话题，看生成的文段是否符合用户的立场和语气。

| 测试项 | 输入 | 结果 |
|--------|------|------|
| 风格匹配度 | 生成的方向是否符合用户的语气和用词 | ✅ 使用括号补充、直接语气、不装 |
| 观点延续性 | 方向是否基于用户之前写过的立场延展 | ✅ 呼应之前对“装”的反感、对本质的追求 |
| 实用性 | 用户拿到方向后能否直接开始写 | ✅ 用户可以接着补自己的经历和想法 |
| 遵循"最终决定你做" | 是否给了用户选择空间而非强制方向 | ✅ 明确说明可以选择继续或换方向 |
