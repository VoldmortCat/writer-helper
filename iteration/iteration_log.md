# AI 写作搭子 — 迭代升级说明

---

## 迭代1：文本类型自动检测（预迭代 — 自发现）

### Step 1 — 描述具体痛点

用第一版 review 分析"学习/绩点/竞赛"那篇经验分享时，"人物"维度完全用不上，
因为议论文里没有人物的概念。分析时只能手动把"人物"替换成"论点与论据"，
但这意味着每次分析议论文都要人工处理一个废掉的维度。

### Step 2 — 量化影响

- 使用场景中有约 50% 是非叙事类文本（议论文、经验分享、公众号文章、课程作业）
- 面对这些文本，"人物"维度给的反馈全是无效的（"本文没有传统人物"）
- 用户需要等 AI 跑完 5 个维度，其中 1 个维度白跑，浪费 20% 的 token 和等待时间

### Step 3 — 假设原因

初始设计时默认用户写的是小说/故事类内容，所以 5 个维度都围绕叙事展开。
但实际使用中，用户也大量写议论文、经验分享、公众号推文——这些文本的核心是
"观点+论据"而不是"人物+情节"。分析框架没有适配不同文本类型的能力。

### Step 4 — 实现方案

在 SKILL.md 的快速审稿流程中增加文本类型自动判断逻辑：

1. **判断依据**：Agent 根据文字特征自动识别
   - 叙事类特征：有对话、人称代词、情节推进词
   - 议论文类特征：有观点词、论证结构（首先/因此/综上所述）
   - 混合类：两类特征都有
2. **两套分析维度**：
   - 叙事类 → 结构 / 人物 / 细节 / 节奏 / 语言
   - 议论文类 → 结构 / 论点与论据 / 细节 / 节奏 / 语言
3. **分析框架**：写入 references/写作技巧参考.md，供 Agent 在分析时参考

### Step 5 — 评估效果

- 同一篇议论文，旧版输出"本文没有传统人物"（无效），新版自动识别为议论文，
  "论点与论据"维度指出"依琳师姐没展开"（有效）
- 叙事类文本不受影响，仍使用原五维度
- 超短文本（<100 字）默认按议论文处理
- ⚠️ 潜在问题：复杂文本（如夹叙夹议的散文）可能误判，需要后续在 prompt 中
  加入 AI 自我确认机制

## 迭代2：一致性检查 — AI检查前后矛盾
【对应 xlsx 迭代1：AI能力增强】

### Step 1 — 描述具体痛点

写作搭子的核心功能之一应该是检查长篇写作的一致性——人物设定有没有矛盾、
前面埋的伏笔后面收没收、时间线有没有冲突。
但当前只有单段反馈，写长篇时作者自己很难发现前文和后文的矛盾，手工翻找极其耗时。

### Step 2 — 量化影响

- 写超过 3 章后，手动回溯检查一章需要 10-15 分钟
- 长篇写完才发现人物设定矛盾（如第一章黑发第五章棕发），
  需要整章重写，代价巨大
- 伏笔忘记回收是长篇写作最常见的 Bug，作者自己很难注意到

### Step 3 — 假设原因

初始设计只考虑了"单篇反馈"场景，没有考虑"跨章节检查"。
review 模式每次只给一段文字反馈，无法感知多章之间的对照关系。
需要一个新的检查维度——不是分析写得好不好，而是对比各章之间有没有矛盾。

### Step 4 — 实现方案

在 SKILL.md 中新增"长篇体检"流程，定义 4 类检查项：

| 检查项 | 说明 | Agent 指令 |
|--------|------|-----------|
| 🚨 人物设定矛盾 | 同一角色在不同章节中外貌/性格/能力/关系有无不一致 | 逐章提取人物特征表，交叉对比 |
| ⚠️ 情节逻辑漏洞 | 时间线冲突、因果关系断裂 | 梳理时间顺序，标记断裂点 |
| 📌 伏笔回收 | 前文埋的线后面收了没有 | 列举已回收 / 未回收伏笔 |
| ✅ 细节照应 | 地名/人名/物品描述前后写错 | 提取关键名词列表，跨章比对 |

Agent 在长篇体检中读取用户提供的所有章节，逐一扫描后输出检查报告，
并标注每条问题的严重程度。最后问用户"要针对这些问题修改吗？"

### Step 5 — 评估效果

测试方式：用真实文章选段做人工验证

| 检查项 | 测试文本 | 结果 |
|--------|---------|------|
| 检查流程可用 | "宗教效应" + "艺术的见解"对比 | 四类检查项输出完整 ✅ |
| 建议实用性 | 用户根据 review 反馈修改"宗教效应" | 4 条修改建议全部采纳 ✅ |

**实证数据：** 用户用「宗教效应」议论文测试 review→修改→对比流程

| 对比项 | 改前 | 改后 |
|--------|------|------|
| 概念定义 | 无定义锚定，直接讲机制 | 开篇锚定「我称它为宗教效应，不是比喻」 |
| 论证结构 | 「实质：」一段塞满 4 个环节 | 拆成 3 段：造物→宣发滚雪球→资源反哺 |
| 语言准确度 | 3 处语病（再/优越/诚如） | 全部修正 |
| 结尾 | 突然截止在「归属感」 | 追加宗教 vs 消费品的对比性结论 |
| 字数 | 608 字 | 793 字 |

✅ 用户反馈：4 条修改建议全部采纳，修改后文章逻辑更清晰、结尾更有力。
证明 review→修改→对比的打磨流程对真实写作有可验证的改进效果。

## 迭代3：风格学习 — 学你的写作风格，保持一致性
【对应 xlsx 迭代2：多能力组合】

### Step 1 — 描述具体痛点

写了一段时间后发现一个问题：同一个人的文章，前后风格不一致。
比如平时用短句、爱括号补充、敢用狠词的人，换了场景写东西就变成套话长篇，
读起来不像一个人写的。AI 默认输出也容易有"AI 腔"，
需要手动调整才能保持个人风格。

### Step 2 — 量化影响

- 写新类型文章时（如议论文->经验分享），风格容易跑偏
- 别人读你的文章会觉得"不像你写的"
- 每次都要手动回顾旧文找风格参考，耗时且不系统

### Step 3 — 假设原因

没有工具记录和对比写作风格特征。风格是一种感觉，
没有量化手段去判断"这句是不是我写的"。

### Step 4 — 实现方案

在 SKILL.md 中增加风格相关指令，并创建 style/ 目录持久化风格记录：

1. **风格提炼流程**：用户给一篇旧文 -> Agent 分析用词偏好、句式特征、语气态度、
   修辞习惯 -> 写入 style/风格记录_用户.md
2. **风格应用场景**：
   - 卡壳疏导：参考风格记录，给出符合用户语气的发展方向
   - 长篇体检：对比文章用词、语气、句式是否跑偏
3. **风格记录结构**（style/风格记录_用户.md）：
   - 用词偏好（爱用什么词、避免什么词）
   - 句式特征（长句多还是短句多）
   - 语气态度（敢用狠词、不装、直接）
   - 修辞习惯（括号补充、反问、排比）

### Step 5 — 评估效果

测试方式：用户提供 4 篇参考文章 -> 风格提炼 -> 用风格判断非本人文章

| 测试项 | 输入 | 结果 |
|--------|------|------|
| 风格提炼 | 宗教效应 + 艺术的见解 + 网友见解 + 个人感想 | 提炼 8 个维度，保存为 style/风格记录_用户.md ✅ |
| 风格识别 | "凌晨三点的便利店"（非本人） | 正确判断"不是用户写的"，标注具体差异：括号使用、口语狠词、语气 ✅ |
| 风格写作 | 用用户风格写"艺术之争总览" | 完成，含括号补充、口语词、直接态度 ✅ |

### 与迭代2的关系

迭代2（一致性检查）管"写对了没有"，迭代3（风格学习）管"像不像你写的"。
两个维度互补。在实际使用中，先跑快速审稿改内容，再跑长篇体检保风格。

## 迭代4：个性化灵感 — 基于你之前写的内容给方向建议
【对应 xlsx 迭代3：个性化闭环】

### Step 1 — 描述具体痛点

卡壳疏导给的方向是通用的——"你可以往左走、往右走、往回走"，
但这些方向跟用户的写作历史、立场、风格没有关系。
用户想要的是"如果你是我，以我过去的观点和风格，这个题我会怎么写"。

### Step 2 — 量化影响

- 卡壳疏导的 3 个方向每次都要用户自己判断"哪个方向适合我"
- 缺乏个性化的方向建议，用户拿到后还要自己再加工
- 风格记录（迭代3产出）和之前的写作存着但没有被用来生成内容

### Step 3 — 假设原因

卡壳疏导设计时没有接入用户的历史写作和风格记录，
每次都是独立给出通用方向。

### Step 4 — 实现方案

在 SKILL.md 的卡壳疏导流程中增加多步联动指令：

1. 用户给话题或卡壳场景
2. Agent 读取 style/风格记录_用户.md（迭代3产出）
3. 回顾用户之前写的相关文章（data/ 中已有）
4. 分析用户之前的立场、关注的话题、论证风格
5. 给 2-3 个符合用户风格和立场的发展方向，每个方向附带一段示例开头
6. 用户选一个接着写，或换方向
7. 用户顺着写出来后，自然切换到打磨跟单流程

### Step 5 — 评估效果

| 测试项 | 输入 | 结果 |
|--------|------|------|
| 风格匹配度 | 生成的方向是否符合用户的语气和用词 | 使用括号补充、直接语气、不装 ✅ |
| 观点延续性 | 方向是否基于用户之前写过的立场延展 | 呼应之前对"装"的反感、对本质的追求 ✅ |
| 实用性 | 用户拿到方向后能否直接开始写 | 用户可以接着补自己的经历和想法 ✅ |
| 遵循"最终决定你做" | 是否给了用户选择空间而非强制方向 | 明确说明可以选择继续或换方向 ✅ |

## 迭代5：流程化跟单 — 从"一次性指令"到"全程跟单"

### Step 1 — 描述具体痛点

原先的 skill 是功能罗列式设计——用户说"帮我看看"就跑 review，说"写不下去"就跑 brainstorm，
每次交互都是一次性交易。用户必须自己知道"下一步该用哪个功能"，Agent 不会主动引导流程。
而且 review/brainstorm/consistency 等功能如果依赖独立脚本调外部 API，
在 Agent 语境下完全是冗余的——Agent 本身就是 LLM，分析能力直接在 prompt 指令中完成，
不需要中间脚本层。

### Step 2 — 量化影响

- 用户每次使用都要手动选择模式，不知道流程怎么串
- 改完一版不会自动对比，用户要再操作才知道改得怎么样
- 外部观点（网友见解、感想）只能当附录贴，没法被 Agent 自动融合到分析里
- 脚本调 API 的方案还多了 API Key 配置的麻烦

### Step 3 — 假设原因

SKILL.md 的正文写成了"脚本功能介绍"而非"Agent 执行指令"。
Agent 读到的是一份工具说明书，不知道在什么场景下该做什么、问什么、下一步是什么。
所有分析逻辑都外包给了外部脚本，没有发挥 Agent 自身作为 LLM 的判断力。

### Step 4 — 实现方案

1. **SKILL.md 正文重构**：从"脚本说明书"改为"流程指令"，定义 4 个跟单流程：
   - 快速审稿（一次性反馈，不跟踪）
   - 打磨跟单（review -> 用户改 -> 对比 -> 再改，改到用户说停）
   - 卡壳疏导（给方向 -> 选方向 -> 给开头 -> 自然切打磨）
   - 长篇体检（一致性+风格检查 -> 出报告 -> 问修不修）

2. **加触发判断表**：Agent 根据用户的第一句话自动选流程，不用用户说"我要用 review 模式"
   - "帮我看看这段" -> 快速审稿
   - "帮我改改" -> 打磨跟单
   - "写不下去了" -> 卡壳疏导
   - "帮我检查整篇" -> 长篇体检

3. **加检查点**：每个关键节点停下来问用户（"要改吗？"、"还要调吗？"），不替用户做决定

4. **加流程切换表**：用户可以在任意时刻跳流程，不强制走完

5. **删除冗余脚本**：原来依赖外部 API 调用的脚本在 Agent 语境下冗余，删除。
   分析能力由 Agent 直接通过 SKILL.md 指令完成，零额外依赖。

### Step 5 — 评估效果

| 测试项 | 输入 | 结果 |
|--------|------|------|
| 快速审稿 | "帮我看看这段" | 5 维度分析 + 指出最需改的问题，一次性不追问 ✅ |
| 打磨跟单 | "帮我改改" | review -> 用户改后对比 -> 逐节分析，改到满意为止 ✅ |
| 卡壳疏导 | "怎么写下去" | 出骨架结构 + 嵌入策略 + 改写建议，用户产出修改版 ✅ |
| 长篇体检 | "帮我检查整篇" | ⏳ 待测（需要更多章节数据） |
| 流程切换 | 打磨中穿插"给点方向" | 从快速审稿->结构建议->打磨，用户主动跳流程顺畅 ✅ |
| 无脚本运行 | 全程不调外部脚本 | 脚本已删除，纯 Agent 指令 ✅ |

## 迭代概要

| 迭代 | 来源 | 核心变化 | 实现形式 |
